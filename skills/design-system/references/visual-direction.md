# Intentional Visual Direction

Load this reference only when creating or materially reshaping a visual direction. Keep extraction and maintenance work faithful to the accepted project evidence unless the user explicitly requests a redesign.

## Ground The Direction

Name the product or subject, audience, surface, and single user job before choosing aesthetics. Derive visual language from the subject's real materials, instruments, data, environment, and vocabulary. Do not fill missing product facts with invented capabilities or generic dashboard content.

Make every major choice traceable to one of:

- verified product or user evidence;
- an accepted project profile;
- an explicitly scoped reference with `use` and `ignore` rules;
- a clearly labeled proposal awaiting approval.

## Build One Coherent Thesis

Define a compact direction before producing artifacts:

- **Palette:** four to six named semantic colors with accessible roles, not decorative swatches.
- **Typography:** deliberate display, body, and utility/data roles using fonts the project can actually load.
- **Layout:** information hierarchy, density, scroll and focus owners, responsive behavior, and an optional small ASCII wireframe when alternatives need comparison.
- **Material:** solid, translucent, native, or mixed surfaces with explicit limits.
- **Motion:** one orchestrated moment or a restrained interaction system only when motion clarifies state, hierarchy, or subject.
- **Signature:** one memorable element rooted in the product's work; spend visual boldness there and keep supporting surfaces disciplined.

Structure must encode meaning. Numbering represents a real sequence, dividers separate actual groups, labels name recognizable user concepts, and emphasis follows task priority. Do not use structural devices merely as decoration.

Match execution complexity to the direction. Maximalism requires enough implementation budget to remain coherent; minimalism requires precise spacing, typography, states, and details. Record the budget and reject a direction the project cannot execute reliably.

## Reject Interchangeable Defaults

Before accepting the direction, ask whether its palette, typography, layout, and signature could be pasted unchanged into an unrelated product. Revise every interchangeable choice.

Treat frequently generated combinations as proposals requiring evidence, not automatic bans. Examples include warm cream with a high-contrast serif and terracotta accent, near-black with one fluorescent accent, generic glass dashboards, and dense editorial grids with ornamental numbering. Use one only when the brief or product evidence genuinely supports it.

Take at most one justified aesthetic risk. Remove decoration that does not improve meaning, recognition, navigation, or task completion.

## Treat Words As Interface Material

- Write from the user's side of the screen and name concepts they recognize.
- Prefer specific active actions such as `Save changes` over generic labels such as `Submit`.
- Keep action names consistent through controls, progress, success, and error states.
- Make errors explain what happened and what the user can do next.
- Make empty states direct the next valid action instead of supplying mood or filler.
- Use real or explicitly marked sample content; do not hide weak information architecture behind polished placeholder copy.

## Apply The Quality Floor

Every proposed direction must account for:

- responsive behavior at the target sizes;
- visible keyboard focus and semantic interaction states;
- reduced-motion behavior;
- loading, empty, error, populated, and permission states when applicable;
- contrast, legibility, overflow, localization, and realistic content density;
- reuse of existing component and token owners before creating another one.

Critique the direction once before artifact creation and again during evaluation. Record what changed and why. `design-system` owns the direction and acceptance contract only; route product-source edits to `implement-frontend` and implementation findings to `audit-frontend`.
